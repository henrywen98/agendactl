# ADR-0010: 二进制命名 `macctl`；calendar / reminders 拆成两个独立 skill

- 状态：accepted
- 日期：2026-06-25
- 上下文：`apple` 作为命令名差——与公司名直接撞、零功能信息、裸 `apple` 不知所云、对将来扩展（Mail/Contacts/Finder）空泛。Henry 要求重新命名，并要求 skill 拆开做。
- 决策：
  - 二进制命名 **`macctl`**（mac + `*ctl` 约定，如 systemctl/kubectl/launchctl，一眼是「控制 mac 应用」的工具）。`bin/apple` → `bin/macctl`。
  - calendar 与 reminders **拆成两个独立 skill**（`.pi/skills/calendar/SKILL.md`、`.pi/skills/reminders/SKILL.md`），对齐 §6.1「一个 App = 一个 skill」。触发上下文不同（"加个日程" vs "提醒我"）、progressive disclosure 更干净，agent 只加载用到的那个。**共享同一个 `bin/macctl`**。
- 后果：
  - 此前 ADR / 文档（0001–0009）里的 `apple` / `bin/apple` 一律**读作 `macctl`**（历史记录不逐一改写；当前态文档——glossary、ADR 索引——已同步为 macctl）。
  - 两个 SKILL.md 按 CLI-only（[ADR-0002]）写：主接口命令 + 先 `lists`/`calendars` 拿真名 + **不提 JXA / 逃生舱**。
  - 命令读感：`macctl calendar list-events ...` / `macctl reminders create ...`。

## 补记（2026-06-25）：skill 作用域 = 项目级（Henry 明确选定）

- 背景：v1.0 §2.4 放项目级的理由是「skill 耦合你的 list/calendar 名」。v2.0 的 skill **不写死容器名**（让 agent 现场 `lists`/`calendars` 查），该理由已失效——曾考虑提到全局 `~/.pi/agent/skills/`。
- 决策：**仍放项目级** `.pi/skills/`，由 Henry 明确选定（理由从「数据耦合」改为「工具作用域刻意收在项目内」）。
- 后果：agent 只在本项目 active 时加载这两个 skill；SKILL.md 让 agent 用 `bin/macctl`（项目内），不依赖全局安装，**自洽的项目级**。
- 旁路：`/opt/homebrew/bin/macctl` 的全局 symlink 仅为 Henry 手动在终端调 `macctl` 的便利，**不影响 skill 的项目级作用域**（可保留或删除，与 skill 范围正交）。
