# ADR-0007: 底座改用 EventKit（Calendar + Reminders），Notes 留 JXA —— 实测推翻 §2.1 的 JXA 选型

- 状态：accepted（amends [ADR-0001]；对 Calendar 路径部分 supersedes [ADR-0006]）
- 日期：2026-06-25
- 上下文：Henry 回看时一问「你是不是用了 Apple **官方 API** 写 CLI」。审讯发现：此前用的是 AppleScript **脚本字典**（sdef via JXA）——官方但**老的 AppleEvent 接口**；**没用 EventKit**（Calendar/Reminders 的现代第一方框架）。文档 §2.1 当初否 EventKit、选 JXA 的**唯一理由是「AI 生成 JXA 正确率高 / token 省」**，而 [ADR-0002]（CLI-only，agent 永不写代码）**已使该理由彻底失效**——CLI 由 Henry 写一次，「AI 不熟 EventKit」不再是减分项。

## 实测对照（EventKit probe，2026-06-25，macOS 26.3）

| 维度 | JXA 脚本字典（实测） | EventKit（实测） |
|---|---|---|
| 容器 id | `calendarIdentifier` 读不出（-1700 / -10000） | **27ms** 读出 7 个 list 稳定 UUID |
| writable | 属性不存在 | `allowsContentModifications`=true 直读 |
| 全量查询 | `whose` 3–6s / 遍历 17s | `NSPredicate` fetch 393 条 = **411ms** |
| 条目 id | event `uid` 可读、calendar id 不行 | `calendarItemIdentifier` / `eventIdentifier` 稳定 |
| 授权 | Automation 桶 | **Reminders 已 fullAccess(3)**；Events 当前 **writeOnly(4)** 待升级 |
| completion+runloop | — | 数据拉取 `fetchDone:true` ✅，osascript 桥可用 |

## 决策

- **Calendar + Reminders 底座 = EventKit**（`EKEventStore` / `EKCalendar` / `EKEvent` / `EKReminder`）。稳定 id、毫秒级 `NSPredicate` 查询、原生 writable，**绕开 §9 整套 -1708/-1728/-10000/whose 日期坑**。
- **Notes 底座 = JXA**（无公开框架，只有脚本字典）——不变。
- 数据操作实测从 osascript 的 EventKit ObjC 桥即可完成（fetch 411ms，completion+runloop 可用）；**仅 events 的 writeOnly→full 授权握手在 osascript 里别扭**（completion 需 GUI 点击）。

## 后果

- [ADR-0006] 的 Calendar 定位策略（calendar 按 name、event 按 uid via `whose`）**对 EventKit 路径作废**——改用 `EKCalendar.calendarIdentifier` + `EKEvent.eventIdentifier` + `NSPredicate`。其实测发现保留为「为何要换」的证据。
- §2.1 选型理由更新；§8/§9/§10 的 Calendar/Reminders 部分对实现者降权（EventKit 不踩那些坑）。
- **新开议题**：CLI 语言/形态（单 osascript-JXA + EK 桥 / Swift / PyObjC）——待审。
- Henry 需**一次性**把 Calendar 授权从 writeOnly 升到 full（首次本机交互，之后 headless 可用，符合 §11 TCC 模型）。
- Notes（JXA）与 Calendar/Reminders（EventKit）**底座分裂**是有意取舍：agent 是 CLI-only（[ADR-0002]），看不到底座差异；复杂度只在 Henry 维护侧。
