# ADR-0006: Calendar 定位策略 —— calendar 按 name + 重名消歧，event 按 uid（实测驱动）

- 状态：accepted；**EventKit 路径下定位部分被 [ADR-0007] 取代**（calendarIdentifier/eventIdentifier + NSPredicate 替代 name/uid+whose）。本 ADR 的实测发现保留为「为何从 JXA 脚本字典转 EventKit」的证据。
- 日期：2026-06-25
- 上下文：按 Henry 确认的方法「先抽 sdef + live probe 校验真实行为，再写 CLI」。实测（macOS 26.3 Tahoe，2026-06-25）推翻 / 补充了 §8.2 的拍脑袋 schema——其中 `calendarIdentifier` 读不出、whose 3–6s 等坑，正是后来 [ADR-0007] 改用 EventKit 的直接动因。

## 实测发现

| 项 | 结果 | 出处 |
|---|---|---|
| 日历数 | **10 个**（§1.2 写 7，已漂移），且**重名**："Australian Holidays" 出现 2 次 | live probe |
| `calendar.calendarIdentifier` | **读不出**：JXA `-1700 Can't convert types`、AppleScript `-10000` | live probe |
| `calendar.writable` | **属性不存在**（sdef calendar 只有 `name`/`title`/`color`/`description`）——§8.2 `.writable()` 是错的 | sdef + probe |
| `event.uid` | **可读**（JXA & AppleScript 均可），且 **create 后立即可读** | live probe |
| 按 uid 反查 | `whose({uid})` **可靠** ~3.3s（uid 是 text，绕开日期坑）；遍历匹配 ~17s（慢 5×） | CRUD round-trip |
| 删除语义 | **硬删**（delete 后 `whose` matches=0），非软删 | CRUD round-trip |
| save 校验 | **强校验 `start < end`**，违反返回 `NSLocalizedDescription` | CRUD round-trip |
| 时区 | 本地 14:00 → `toISOString()` = `06:00Z`（UTC+8），§10.5 序列化成立 | CRUD round-trip |

## 决策

- **日历定位用 `name`**；重名时退出码 2 + 列出候选（或 `--index` 消歧）。**无稳定 id 可用**——`calendarIdentifier` 两条路都读不出。
- **事件定位用 `uid`**；`create-event` 返回 uid；`update-event <uid>` / `delete-event <uid>` 走 `whose({uid})` 反查（**禁止遍历**）。
- CLI 参数校验**强制 `end > start`**（改时间时若只动一端要保证顺序仍合法）。
- `apple calendar calendars` **不报「可写性」**（属性不存在）；可写性靠 `create-event` 失败时的错误兜底。

## 后果

- 驱动 `apple calendar` 命令签名与 `lib/` 的 Calendar helper。
- 更新 glossary 的 `id` / `name` / `writable` / `Container name 解析` 条目（见 glossary.md）。
- 每次 uid 操作 ~3–6s（在 ~878 事件里 whose 扫描），对个人自动化可接受；未来可用 `--calendar` 收窄优化。
- 原 sdef 快照存 `references/calendar.sdef.xml`（§12 快照策略），macOS 升级时 diff。
